# AHUTong 协作规范

本文件代表 AHUTong 项目的一些协作规范。所有参与者在修改本仓库前都应先阅读并遵守这些约定。

## 分支规范

1. 需要编辑文件时，必须先创建属于自己的工作分支，不要直接在主分支上修改。
2. 分支命名格式：
   - 功能开发：`p/{你的 GitHub 用户名}/feat`
   - 问题修复：`p/{你的 GitHub 用户名}/fix`
3. 开始工作前建议先同步目标基线分支，避免在过期代码上开发。
4. 一个分支尽量只处理一个明确目标，避免把无关功能、重构和格式化混在一起。

## 项目本地 Skills

1. Session start 时应检查 `.agents/skills/*/SKILL.md` 的 frontmatter，了解项目本地可用 skill。
2. `.agents/skills/yukon-auto-update-release/` 是生产发版 skill；只有当用户请求中显式包含“发版”两个字时才允许触发或执行。
3. 未包含“发版”的构建、打包、上传、版本检查或 release 相关请求，都不得使用 `yukon-auto-update-release`。

## 提交规范

1. 提交信息必须符合 Conventional Commits 规范。
2. 常用格式：

   ```text
   <type>(optional scope): <description>
   ```

3. 常用 `type` 包括：
   - `feat`: 新功能
   - `fix`: 问题修复
   - `docs`: 文档变更
   - `style`: 不影响逻辑的格式调整
   - `refactor`: 重构
   - `perf`: 性能优化
   - `test`: 测试相关
   - `build`: 构建系统或依赖变更
   - `chore`: 其它维护性变更
4. 本地存在多个待推送 commit 时，push 前应先 squash，保持提交历史清晰。
5. 示例：

   ```text
   feat(schedule): add week selector
   fix(login): handle expired session
   docs: update contribution rules
   ```

## 修改与审查规范

1. 修改前先确认当前工作区状态，不要覆盖或回退他人的未提交改动。
2. 保持改动范围聚焦，只提交与当前任务直接相关的文件。
3. 涉及用户可见行为、数据读写、登录态、缓存、网络请求或原生库的改动，应优先补充说明和必要测试。
4. 提交前至少运行与改动相关的构建、检查或测试；如果无法运行，需要在提交说明或合并请求中写明原因。
5. 不要提交本地配置、密钥、账号信息、构建产物或临时调试文件。
6. 对公共接口、数据结构、权限、依赖版本和 Gradle 配置的修改，需要在合并请求中说明影响范围。
7. 开发时如果用户同时提出多个需求或多个 bug 修复，默认允许开启 subagent 进行拆解、并行分析或处理。

## 编译与运行验证

1. 编译测试优先使用项目的 Gradle 命令，确保结果可复现。
2. 每次准备进行编译测试时，应先通过 `adb devices` 检测是否存在可用设备。
3. 检测到可用设备时，应执行与 Android Studio 点击运行相近的流程：构建 Debug 包、安装到设备并启动应用。
4. 未检测到可用设备时，才只执行编译、检查或单元测试，不尝试安装或启动应用。
5. 运行验证完成后，应在提交说明或合并请求中写明使用的命令、设备状态和验证结果。

## 代码风格

1. 遵循项目现有的 Kotlin、Java、Compose 和 Gradle 写法，不为单个改动引入不必要的新框架或新抽象。
2. 命名应表达业务含义，避免无意义缩写。
3. 注释用于解释必要背景、约束或复杂逻辑，不重复代码表面含义。
4. 格式化应尽量局部进行，避免无关的大规模重排。

## 合并前检查

1. 确认分支名称符合规范。
2. 确认提交信息符合 Conventional Commits。
3. 确认没有混入无关文件、敏感信息或本地环境文件。
4. 确认关键功能路径已经手动验证或通过自动化测试验证。
5. 合并请求中应简要说明改动内容、验证方式和可能影响。
